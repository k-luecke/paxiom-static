§ A · Strategic Position

Vendor, not counterparty.

The earlier framing treated cross-chain arbitrage as the primary revenue path with the proof platform as supporting infrastructure. That ordering is wrong for this operator and this household. Arbitrage is capital-intensive, directionally exposed, competes against teams with structural advantages, and produces volatile income that does not sit well next to existing extended-family obligations.

The reframe: Paxiom is a HyperBEAM-native distributed proof orchestration platform with x402-gated public endpoints as the primary revenue surface. Cross-chain settlement becomes the reference application that demonstrates the platform works. Arbitrage P&L from operating that reference is bonus margin, not the business.

You collect transaction fees on infrastructure other people use to take risk. Same shape as AWS, Stripe, Twilio. Different shape from Citadel.

§ B · Why This Reframe

The structural difference.

Arbitrage path

Capital-intensive. Variance is high, tail risk is real. Competes against Jump, Wintermute, GSR, MEV searchers with better latency and deeper relationships. A bad week is a five-figure drawdown. Income volatility doesn't sleep well next to extended-family obligations.

Platform path

Capital-light. Revenue per transaction with positive margin on compute cost. No directional exposure. Bad weeks for traders are still revenue weeks for you because volume goes up when markets are volatile. Customers are agentic systems needing verified compute — a market larger than the 200 industrious individuals.

The category isn't profitable for anyone yet. Whoever finds genuine product-market fit at sustainable unit economics first wins disproportionately. Capital efficiency is the moat.

§ C · Sequencing

Six phases, evaluated at each gate.

Each phase produces enough information to make the next decision. Each phase generates revenue earlier than its replacement would. Failures fail cheaply because nothing downstream depends on them.

00
Foundation · weeks 1–2

Substrate familiarity. No product yet.

  • HyperBEAM running locally via rebar3 shell. If this doesn't work, nothing downstream matters.
  • Read the Load Network HyperBEAM fork source: ~evm@1.0, dev_evm.erl, the AO process examples.
  • Read the AO core protocol docs and hb_ao device source. Understand message flow, ordering, persistence.
  • Lurk on ZK Hack Discord, Arweave/AO Discord, Decent Land community channels. Read for two weeks before posting.
  • Pin the segmentation manifest on Arweave. Smallest first artifact. Locks the protocol shape.
01
Minimum viable proof device · weeks 3–6

One device. One predicate. One number.

  • HyperBEAM device wrapping the existing slot0 proof generation. Single block, hardcoded inputs, end-to-end.
  • Deploy Groth16 verifier on Optimism / Base / Arbitrum Sepolia. Measure exact gas cost on each.
  • Instrument the dispatch path. Log timestamps for witness gen, proof gen, AO coordination, Arweave commitment, on-chain verification.
  • Steady-state staleness budget: define the ceiling per chain explicitly before measuring.
Gate 01

If verification gas exceeds 400k on any target chain, or end-to-end latency exceeds 30 seconds in steady state — the architecture needs adjustment before proceeding. Pivot proof system or accept longer settlement window.

02
x402-gated endpoint · weeks 7–12

First revenue from infrastructure, not trades.

  • Wrap the proof device behind an HTTP server implementing x402. Single proof type, single price, single endpoint.
  • Test with Cloudflare's Agent SDK or equivalent. Verify an off-the-shelf agent can discover, pay, consume.
  • Document the endpoint. Publish to whatever directories x402 services are listed in (early; build relationships with directory maintainers).
  • Run the public endpoint for a month. Measure inbound traffic. Learn what agents actually request.
  • Light arbitrage on own book at small capital ($5–10k) using own platform — reference customer, not primary revenue.
Gate 02

If x402 endpoint generates non-trivial inbound traffic in first month — proceed. If not — the agentic-payment thesis is premature; pivot to direct B2B sales or accept slower customer acquisition.

03
Catalog expansion · weeks 13–20

More proof types. More chains. Simulation as product.

  • Expand proof catalog: range proofs, aggregated proofs, cross-chain state predicates beyond slot0.
  • Add simulation-as-a-service endpoint using the ~evm@1.0 device. Pre-flight check for arbitrageurs and other agents — they pay per simulation.
  • Multi-backend support: route work between RISC Zero, SP1, Plonky3, Orion/Rokokko. Customer picks proof system.
  • State reconstruction integration: use Load Network archive-based reconstruction as default input source. Trust-minimization differentiator vs. competitors depending on Alchemy / Infura.
04
B2B + cross-chain settlement · months 6–12

The reference application becomes the showpiece.

  • Build the source-chain escrow contract on mainnet with the explicit state machine: INIT → LOAN_REQUESTED → COLLATERAL_LOCKED → LOAN_DISBURSED → AWAITING_PROOF → [PROOF_VALID_SUCCESS | PROOF_VALID_FAILURE | TIMED_OUT | LIQUIDATED].
  • Destination-chain borrower contract. Commitment relayer. Full bounded-time settlement loop tested on testnet, then mainnet small.
  • B2B sales motion: approach 5–10 named ZK organizations with concrete operational data. Uptime, gas costs, pricing, what their engineers stop having to do.
  • Each B2B integration is a 3–6 month engagement; first paying customer unlocks reference for next three.
Gate 04

At least one paying B2B customer integrated by end of month 12, or platform productization is happening too slowly relative to household timeline. Adjust scope.

05
Specialization · year 2+

Hardware partnerships. Adjacent markets.

  • Hardware acceleration partnerships once volume justifies negotiation. Cysic, Ulvetanna, Ingonyama. Routing layer picks optimal hardware per workload.
  • TEE-attested proving via HyperBEAM's ~greenzone@1.0 device for customers preferring TEE trust model over ZK.
  • Expansion to non-crypto applications: verifiable AI inference, regulatory compliance, identity. Same capabilities, different customers.
  • Post-quantum migration path becomes a product as NIST 2030 deprecation deadline approaches and competitors face migration headaches.
§ D · Continuous Workstreams

Run across all phases.

Community presence

Lurk now, contribute later. Solve ZK Hack puzzles publicly. Answer technical questions thoughtfully. Build name recognition through substance, not marketing. ETHDenver 2027 is the first major in-person investment.

Documentation

Every device gets clean documentation. Every endpoint a clear specification. Construction PM background should make this natural — it's the same operational discipline applied to software.

Operational rigor

Uptime, monitoring, incident response, postmortems. Every outage is a customer-acquisition setback. Every clean postmortem is a public credibility builder. The platform's value is reliability under load.

Financial track

Track infrastructure cost against revenue from day one. Phases 2 onward should fund continued development without external capital. If those numbers don't materialize on schedule — that's signal. Adjust before runway runs out.

§ E · What this map does not promise

The honest list.

This map does not promise the architecture works. Phases 1 and 2 might reveal a fatal flaw in dispatch latency, gas costs, or the proof system selected. Phase 3 might reveal that x402 adoption is too early for the platform to find customers there. Phase 4 might reveal that B2B sales motion is harder than the public endpoint motion.

What the map does promise: each phase produces enough information to make the next decision. The work compounds rather than starts over. Revenue arrives as early as possible to fund subsequent phases without external capital. Every phase that works funds the next; every phase that fails fails cheaply.

The binding constraint is not the technical roadmap. It's the household timeline. Anything that doesn't move toward "platform generates revenue at increasing rates over the next 18–24 months" is lower priority than things that do.

The architectural elegance of running as a HyperBEAM device matters because it lowers build cost and accelerates the timeline. The x402 integration matters because it expands the customer base beyond what reputation can sell into. The proof platform matters because it generates fee revenue from day one rather than capital-at-risk returns. Every architectural choice in this map serves the timeline, not the other way around.
